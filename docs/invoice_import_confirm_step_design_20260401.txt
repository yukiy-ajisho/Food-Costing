--------------------------------------------------------------------------------
0. 目的・スコープ
--------------------------------------------------------------------------------

- Import invoice モーダルの **Confirm（ステップ 4 相当）** を、集約サマリーではなく
  **行（アイテム）単位の最終反映内容**の一覧に置き換える。
- **Existing** 行について、請求日と VVP 更新時刻の関係で **警告表示**し、
  **右下にスタックした確認ポップアップ**でユーザー承認を取る（該当 0 件なら非表示）。
- **Existing** 行ごとに、**VVP の表示価格を今回の請求で更新するか／維持するか**を
  **ラジオ**で選べるようにする（**New 行は対象外**）。
- **請求日**は OCR（Gemini）で取得し、**Review lines** でブロックごとに**修正可能**とする。

対象外（本書では詳述しないが依存しうる）:
- Match ステップの New/Existing 自動マッチ（別紙の §9 など）。

--------------------------------------------------------------------------------
1. 現状のデータフロー（実装確認済み・ベースライン）
--------------------------------------------------------------------------------

1-1. New（インポート時に新規 VVP を作る行）

- フロント: `vendorProductsAPI.create` → `productMappingsAPI.create`
- バックエンド `POST /vendor-products`（backend/src/routes/vendor-products.ts）:
  - `virtual_vendor_products` に insert（`current_price` 含む）
  - 続けて `price_events` に **1 行 insert**（`price` = 上記 `current_price`、
    `source_type` = `invoice` または `manual`、`initial_price_event_source` に応じる）
- フロントの `recordInvoice` は **ループ内では呼ばない**が、**create 内で price_events が書かれる**。

1-2. Existing（既存 VVP に請求価格を載せる行）

- フロント: `priceEventsAPI.recordInvoice(vvpId, price)`
- バックエンド `POST /price-events/vendor-products/:id/invoice`:
  - `price_events` に **insert** のみ（`source_type` = `invoice` 等）

1-3. DB トリガー（supabase/schema_dump.sql）

- `AFTER INSERT ON price_events` → `sync_virtual_vendor_current_price_from_event`
- 処理内容:
  - `virtual_vendor_products.current_price` ← **NEW.price**（当該 insert 行）
  - `virtual_vendor_products.updated_at` ← **NEW.created_at**（当該 price_event 行の created_at）
- `price_events` テーブルに `updated_at` は想定せず、**VVP.updated_at は「そのイベントの発生時刻」に合わせる**。

1-4. 整理（ユーザー向け説明と整合）

- **通常経路**: `price_events` に 1 行足すと、**例外なく**上記トリガーで **VVP の
  current_price と updated_at が更新される**（Existing の recordInvoice、New の create 後の
  初回イベントの両方）。
- **「表示上の VVP 単価」**は、**最後にトリガーを走らせた price_event の price** と一致する
  （その後、VVP をトリガー外で直接 UPDATE した場合などはずれる可能性あり）。

--------------------------------------------------------------------------------
2. Confirm ステップの画面要件
--------------------------------------------------------------------------------

2-1. 表示内容の置き換え

- **やめる**: 「Create N new… / Record M existing… / By block の行数だけ / File 名」中心の
  集約サマリー。
- **する**: **最終的に反映されるレコードを、アイテム（請求行）の数だけ**行単位で列挙。
  - 列の例: ブロック、請求日、モード（New/Existing）、商品摘要、数量・単位、請求価格、
    反映内容の説明（新規 VVP＋マッピング／既存へのイベント等）、Existing 専用のラジオ列（後述）。

2-2. ブロック（invoice まとまり）ごとの請求日

- **データソース**: PDF から Gemini が抽出する日付（構造化 JSON にフィールドを追加する想定）。
- **Review lines（ステップ 2）**: **各ブロックに請求日を表示・編集できるフィールド**を追加する。
- Confirm では **その値**を「当該ブロックの請求日」として表示し、以降の比較に使う。

--------------------------------------------------------------------------------
3. Existing 行の「古い請求」警告（パステル黄）
--------------------------------------------------------------------------------

3-1. 条件

- 行が **Existing** かつ、紐づく **VVP** について:
  - **請求日（ブロックの日付）** が **VVP.updated_at** より **古い**（厳密: 日付比較ルールは実装時に定義。
    例: 日単位で invoice_date < date(VVP.updated_at) など）
- 該当行の行背景を **パステルイエロー**で強調。

3-2. 意味（UX）

- 「マスタ上、この VVP は請求日より**後**に更新されている可能性がある」＝
  **今回の請求はタイムライン的に古い**可能性の注意喚起。

3-3. 比較対象について

- 第一候補: **`virtual_vendor_products.updated_at`**（フロントが既に VVP 一覧を持つなら利用しやすい）。
- **通常、invoice 経由で「Use invoice price」を続けている限り**、直近の `price_events` insert に
  トリガーが追従するため、**VVP.updated_at と「直近の請求系イベントの created_at」は概ね一致**しうる。
- より厳密に「最終請求イベント日時」だけ見たい場合は、実装時に **price_events の最新 created_at**
  との比較も検討可。

--------------------------------------------------------------------------------
4. 右下ポップアップ（スタック・OK 必須）
--------------------------------------------------------------------------------

4-1. 表示タイミング

- **Match 完了後、Confirm に遷移する「Next」押下後**（ステップ境界は実装で合わせる）。
- **黄強調の行が 0 件**のときは **ポップアップを出さない**。

4-2. 件数・粒度

- **1 件のまとめダイアログではなく**、**黄強調対象の行ごとに**、画面**右下**に
  **スタック（複数段）**で表示するイメージ。
- 各ポップアップに **OK** があり、**すべて OK するまで**次操作（例: Import 実行）に進めない、
  または Import をロックする（詳細は実装で決定）。

4-3. 文言

- 「古い請求の可能性がある」旨と、**該当行の識別情報**（商品名・VVP 表示名など）を含める。
- 文言は英語／日本語はプロダクト方針に合わせる。

--------------------------------------------------------------------------------
5. Existing 行のみ: VVP 単価の扱い（ラジオ）
--------------------------------------------------------------------------------

5-1. 選択肢

- **A — Use invoice price（従来どおり）**
  - **price_events** にレコードを追加する。
  - **既存トリガー**により **VVP.current_price** と **VVP.updated_at** が更新される。

- **B — Keep current VVP price（表示単価は変えない）**
  - **price_events** には **レコードを残してよい**（監査・履歴用途）。
  - **VVP.current_price** および **VVP.updated_at** は **更新しない**（以前のまま）。

5-2. 実装上の注意（重要）

- **現状の DB では、price_events への INSERT のたびに必ず同期トリガーが動く**。
- したがって **B を実現するには**、次のいずれか（または併用）が**新規設計**として必要:
  - トリガーを **条件付き**にする（例: price_events に「VVP を同期しない」フラグ列を追加）
  - **B 専用 API** で insert 後に VVP を**元の値に戻す**（競合・整合性に注意）
  - 別テーブルで「請求のみの履歴」を持つ（スキーマ拡大）
- **フロントのラジオだけでは足りない**。バックエンド／マイグレーションとの**合同タスク**とする。

5-3. New 行

- ラジオ列は **表示しない**（または「—」）。常に **新規 VVP ＋ マッピング ＋ 初回 price_event**
  （現行 create の挙動）。

--------------------------------------------------------------------------------
6. 実装タスクの分割案（参考）
--------------------------------------------------------------------------------

- フロント
  - Review: ブロック単位の請求日（state・OCR マッピング・バリデーション）
  - Confirm: 行一覧 UI、黄強調、Existing ラジオ、スタックトースト、Import ガード
  - Match→Confirm 遷移時のデータ受け渡し（請求日を groups に保持）
- バックエンド／DB
  - B 選択時の **price_events と VVP の分離**（§5-2）
  - 必要なら `price_events.invoice_id` や請求日との紐付け（将来の監査）
- OCR
  - structured_json に **invoice_date またはブロック単位日付**を追加しパース

--------------------------------------------------------------------------------
7. 未決だった事項への回答（決定）
--------------------------------------------------------------------------------

- **請求日と `updated_at` の比較粒度**
  - **日付のみ**で比較する（時刻は使わない）。
  - `updated_at` は DB のタイムスタンプのままでよいが、黄判定では **同一タイムゾーン基準で日付に落とした値**と請求日を比べる（テナント既定 TZ 等は実装で固定）。

- **ポップアップのインタラクション**
  - 黄対象ごとにポップアップ（または同等の確認 UI）を出す。
  - **OK でその 1 件だけ閉じる**。残りがある間は **Import は押せない**（ボタン disabled 等でガード）。
  - 全件について確認が済むまで Import 不可。

- **B の DB 設計**
  - **フラグ列（パラメータ）＋条件付きトリガー**とする。
  - 例: `price_events` に **VVP の current を同期するか**を表す boolean（デフォルト true）。`false` の INSERT では既存の同期トリガーが **VVP を更新しない**。
  - **新規 API だけに頼る方式は採用しない**（INSERT 時にトリガーが必ず走るため、列による分岐が必要）。

- **黄対象が多いとき**
  - **同時に表示する件数の上限は 10**。超える分はキューに保持。
  - **1 件 OK するたびにキューから次の 1 件を表示域に繰り上げる**（常に「表示中 ≤ 10」）。
  - **折りたたみ UI は不要**（キュー＋上限で代替）。
