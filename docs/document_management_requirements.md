# ドキュメント管理機能 要件定義書

## 1. 概要

### 1.1 背景・目的

飲食店運営において、以下のニーズに対応する機能を追加する。

- **属人性のある書類の管理**: 従業員ごとの許可証・資格証・健康診断書などを一元管理し、アップロード・有効期限の把握を行う。
- **お店（テナント）単位の書類の管理**: アルコール販売許可、税務関連書類など、店舗に紐づくドキュメントの保管・有効期限管理を行う。
- **「誰に何が必要か」の可視化**: ロール（役割）ごとに「必要なドキュメント」を定義し、各担当者に対して「この人は何が揃っているか／何が不足しているか」を一覧で見せる。
- **期限・支払の見える化（カレンダー）**: 更新期限や支払日を忘れないよう、カレンダー上でわかりやすく表示する（将来拡張）。

既存の Food Costing アプリはテナント（お店）・ユーザー・ロール（admin / manager / staff）をすでに持っているため、これらを前提にドキュメント管理を「人の管理とドキュメントの管理のセット」として設計する。

**現状の補足**: テナントは運用されている。ロールは存在するが、**staff は現時点では実質未使用**（機能として有効化されていない）とする。ドキュメント機能でも staff 向けの要件・UI は「将来対応」としておき、まずは admin / manager を前提に設計してもよい。

## 2. 既存アプリのどこまで使うか（範囲）

**既存テーブルをそのまま使うもの**

| 既存リソース | 用途 |
|--------------|------|
| **tenants** | ドキュメントの所属先。「お店の書類」は tenant_id のみで紐づく。データのスコープ（`.in("tenant_id", req.user.tenant_ids)`）も既存と同じ。 |
| **users**（auth.users / public.users） | 「誰の」書類か。ドキュメントの user_id は「この人の書類」を指す。新規テーブルは作らない。 |
| **profiles** | ユーザー×テナント×ロール。「この人は何が必要か」を出すときに、その人の (tenant_id, role) を profiles から取得し、ロール別要件と照合する。新規テーブルは作らない。 |

**新規に作るもの**

| 新規 | 用途 |
|------|------|
| **documents** | 書類1件ごと（タイトル・有効期限・種類・紐づき先 user/tenant・ファイル参照など）。既存の items / recipe_lines 等は食品原価用のため流用しない。 |
| **書類の種類マスタ** | 後述「5. 書類の概念を登録するか」を参照。 |
| **ロール別必須要件** | 「このロールにはこの種類が必須」という定義。profiles には「種類」がないので新規テーブル。 |

**使わない既存テーブル**

- items, recipe_lines, base_items, labor_roles, vendors, virtual_vendor_products, product_mappings などは **食品原価ドメイン専用**。ドキュメント機能では参照しない。

## 3. 用語・前提の整理

| 用語 | 意味（本要件での定義） |
|------|------------------------|
| **テナント** | 1店舗＝1組織。既存の `tenants` テーブル。type は `restaurant` / `vendor`。 |
| **ユーザー（人）** | 既存の `users`（auth 連携）。1ユーザーが複数テナントに属しうる。 |
| **プロファイル** | ユーザーが特定テナントに属するときの「所属＋ロール」。既存の `profiles`（user_id, tenant_id, role）。 |
| **ロール** | 既存と同様 `admin` / `manager` / `staff`。**注**: staff は現状未使用。要件はロール単位で定義する。 |
| **ドキュメント** | 1ファイル（または1件の登録）を表す。タイトル・有効期限・紐づき先（ユーザー or テナント）を持つ。 |
| **リクアイアメント（要件）** | 「このロールには、この種類のドキュメントが必須」という定義。ドキュメントの「種類」を表すマスタ的な存在。 |

**社長メモの解釈補足**

- 「各属人性」→ 従業員（ユーザー）に紐づく書類（許可証・資格など）。
- 「パーミット」→ 許可証類。ドキュメントの一種として扱う。
- 「アカウントにリンク」→ ドキュメントを「特定ユーザー」または「テナント（お店）」に紐づける。
- 「この人は何が必要か」→ あるユーザー（＋そのテナント内ロール）に対して、「必須ドキュメント種類」のうちどれがアップロード済みでどれが未提出か」を表示する。
- 「二つをくっつけるのはテーブルじゃなくていい、クエリでできる」→ ドキュメントテーブルと「ロール別必須要件」テーブルを別に持ち、**「この人に必要なもの」はクエリ（JOIN＋条件）で算出する**方針とする。

## 4. 権限の方針（既存アプリに合わせるか）

**方針案: 既存アプリと同様に「admin が全体を握り、一部を manager に譲渡」**

- **admin**
  - そのテナントのドキュメント・書類種類マスタ・ロール別要件のすべてを閲覧・作成・編集・削除できる。
  - 誰の書類でもアップロード・編集・削除可能。お店用書類も同様。
- **manager**
  - 既存の Food Costing と同様、**admin が明示的に許可した範囲**で操作できるようにする。
  - 実装オプション:
    - **A**: ドキュメントは「テナント内なら誰のものでも manager が編集可」（シンプル）。
    - **B**: 既存の `resource_shares` / `responsible_user_id` のような「譲渡」概念をドキュメントにも持ち、admin が「この書類は manager に編集権を渡す」などと設定できるようにする（将来拡張）。
  - 初期は **A** で進め、必要になったら B を検討するのが現実的。
- **staff**
  - 現状 staff は機能していないため、ドキュメント機能でも **初版では staff 用の画面・権限は作らない** 想定でよい。
  - 将来 staff を有効化するときは、「自分の書類のみ閲覧・自分に紐づく書類のアップロードのみ可」などと定義する。

**認可の実装**: 既存と同様、Cedar の Principal（tenant_id, role）と Resource（owner_tenant_id 等）で評価する。ドキュメント用の resource_type（例: `document`）を追加し、ポリシーで admin / manager / staff の動作を分ける。

## 5. 書類の「概念」（種類）を登録できるようにするか

**社長の話に含まれていたか**

- 社長メモには「各一人一人に対してリクアイアメントをつける（roleごとに）」とあり、**ロールごとに「何が必要か」を定義する**ことは明示されている。
- 「書類の種類そのもの（健康診断・調理師免許・アルコール許可など）を、アプリ上で登録できるようにするか」は、**メモには直接は書かれていない**。ただし「パーミット」「アルコールや税金のドキュメント」など**種類の例**は挙がっている。

**推奨: 書類の概念（種類）は登録できるようにする**

- **理由**
  - 店舗ごとに必要な書類が違う（例: 酒類を扱わない店はアルコール許可不要、税務書類の種類も店による）。
  - 法改正や運用変更で新しい種類（例: 防火責任者講習修了証）を後から増やしたいニーズがある。
- **実装**
  - **書類種類マスタ**をテナント単位で持つ（例: `document_types` テーブル）。属性: 名前、ユーザー用かテナント（お店）用か、表示順など。
  - ドキュメント1件は「どの種類に属するか」をこのマスタへの FK で持つ。ロール別必須要件は「このロールには、この document_type が必須」と定義する。
- **初期リリースで全部作らなくてもよい**
  - まずは「固定の数種類（健康診断・調理師免許・アルコール許可・税務書類など）だけ Enum で持つ」でも動く。運用してみて「種類を増やしたい」が出たら、マスタ化（登録機能）を追加する、という段階でもよい。

**結論**: 社長の話には「種類の登録」は明示されていないが、目的に合うので**書類の概念（種類）は登録できるようにする設計を推奨**する。フェーズとしては、Phase 1 で固定種類で始め、Phase 2 で「種類の CRUD」を入れてもよい。

## 6. 機能要件

### 6.1 ドキュメントの管理

- **アップロード・登録**
  - ファイル（PDF/画像等）のアップロード、または「外部URL・メモのみ」の登録を想定。
  - 1件ごとに以下を持つ：
    - **タイトル**（必須）
    - **紐づき先**
      - **ユーザーに紐づく**: そのテナント内の「誰の」書類か（user_id + tenant_id で一意に特定）
      - **テナントに紐づく**: お店の書類（アルコール許可・税務書類など）→ tenant_id のみ、user_id は null
    - **有効期限**（任意。null の場合は「期限なし」）
    - **ドキュメント種類（requirement_type 等）**: 後述の「ロール別リクアイアメント」と対応づけるための分類（例: 健康診断、調理師免許、アルコール販売許可、税務書類）。
- **一覧・検索・編集・削除**
  - テナント単位で、そのテナントに紐づくドキュメントを一覧表示。
  - フィルタ: 紐づき先（ユーザー別 / お店用）、種類、有効期限の有無、期限切れなど。
  - 編集: タイトル・有効期限・種類の変更。ファイルの差し替え（再アップロード）も想定。
  - 削除: 論理削除または物理削除（ポリシーで決定）。

### 6.2 リクアイアメント（必須ドキュメント要件）の管理

- **ロールごとの必須要件**
  - 「このロール（admin / manager / staff）には、この種類のドキュメントが必須である」を定義するマスタを用意する。
  - データとしては「テナント × ロール × ドキュメント種類」で、「そのロールの人は、その種類のドキュメントを1つ以上提出していること」を要求する。
- **テナント単位の「お店用」要件**
  - 「このテナントでは、アルコール許可証・税務書類などを保持していること」といった要件も、種類として定義し、テナントに紐づくドキュメントで満たす。
- **運用**
  - リクアイアメントの種類（例: 健康診断、調理師免許、アルコール許可、税務書類）は、テナントごとに「どのロールにどの種類を要求するか」を設定できるようにする（将来、テナントごとにカスタム種類を増やす拡張も考慮可能）。

### 6.3 アウトプット:「この人は何が必要か」

- **画面・API**
  - 対象: テナント内の「人」（ユーザー）。そのユーザーがそのテナントで持つロール（1ユーザーが1テナントに1ロール想定）を参照する。
  - 表示内容:
    - そのロールに要求されている「ドキュメント種類」一覧。
    - 各種類について、「該当するドキュメントが提出済みか」「有効期限はいつか」「期限切れか」を表示。
  - 結果として「この人は何が必要か」（不足している要件のリスト）と「揃っているもの」の両方を示す。
- **実装方針**
  - ドキュメントテーブルとリクアイアメント定義テーブルは別。**「この人に必要なもの」はクエリで結合して算出**する（中間テーブルは作らず、ロール＋種類で必要な要件を取得し、ドキュメントの有無・有効期限を JOIN/サブクエリで判定する）。

### 6.4 お店（テナント）のドキュメント

- アルコール販売許可、税務関連書類などは「テナントに紐づくドキュメント」として登録する。
- リクアイアメント側で「テナント用」の種類（例: アルコール許可、税務書類）を定義し、「テナントとしてこの種類が1件以上あること」を満たしているかを同じ仕組みでチェックできるようにする。
- 一覧画面では「従業員用」と「お店用」をタブやフィルタで切り替え可能にするとよい。

### 6.5 カレンダー機能（将来）

- **目的**: 有効期限や支払日を「ちゃんと払ったか」「更新が必要な時期」を忘れないように、わかりやすく見えるようにする。
- **想定内容**
  - 有効期限の近いドキュメントを日付軸で表示（カレンダーまたはリスト）。
  - オプション: 支払日・更新手続き予定日など、ドキュメントに「次回アクション日」を持たせ、カレンダーに表示する。
- フェーズとしては「ドキュメント管理＋リクアイアメント」の後に、カレンダー表示・リマインダーを追加する形を推奨。

## 7. データ要件（テーブル設計方針）

※ 実装時の詳細スキーマ（カラム名・型・インデックス）は別途設計する。ここでは「何を格納するか」の要件レベルで記載。

### 7.1 ドキュメントを格納するテーブル（案: `documents`）

- **目的**: 1件のドキュメント（ファイルまたは登録情報）を表す。
- **主な属性**
  - 識別子（id）
  - テナントID（tenant_id）— 必須。どの店のデータか。
  - 紐づき先:
    - ユーザーに紐づく場合: user_id（NOT NULL）, テナント内の「誰の」書類かは user_id + tenant_id で判断。
    - お店に紐づく場合: user_id（NULL）。テナントの書類。
  - タイトル（title）
  - ドキュメント種類（requirement_type_id または種類コード）— 後述の「リクアイアメント種類」と対応。
  - 有効期限（expires_at, 任意）
  - ファイル格納: ファイルパスまたはストレージ（S3/Supabase Storage 等）のキー、または URL。ファイルなしの「メモのみ」の場合は null 可。
  - 作成日時・更新日時・作成者（user_id は「誰が登録したか」と「誰の書類か」で別なので、created_by など必要に応じて）。
- **制約**
  - テナントに属するデータは、そのテナントのメンバーしか触れない（既存の認可・テナント分離と整合）。

### 7.2 「誰に何が必要か」を定義するテーブル（案: `document_requirements` または `role_requirement_types`）

- **目的**: ロールごとに「この種類のドキュメントが必須」を定義する。**ドキュメント自体のテーブルとは別**。
- **主な属性**
  - 識別子（id）
  - テナントID（tenant_id）— テナントごとに「どのロールに何を要求するか」を変えられるようにする。
  - ロール（role）: admin / manager / staff のいずれか。
  - ドキュメント種類（requirement_type_id または種類名）: 例: 健康診断、調理師免許、アルコール許可（テナント用）。
  - 「テナント用」フラグ: その種類が「ユーザーに紐づく書類」か「テナントに紐づく書類」かを区別。
- **「この人は何が必要か」の算出**
  - このテーブルには「人（user_id）」は持たない。
  - クエリで: 対象ユーザーの profile（tenant_id, role）を取得 → その tenant_id + role に紐づく requirement 一覧を取得 → 各種類について、そのユーザー（またはテナント）に紐づく documents の有無・有効期限を JOIN/サブクエリで判定。
  - つまり「人とドキュメントをくっつける」のは**中間テーブルではなくクエリ**で実現する、という方針。

### 7.3 ドキュメント種類のマスタ（書類の概念）

- 種類の定義: 健康診断、調理師免許、アルコール販売許可、税務書類 など。
- **推奨**（本文「5. 書類の概念を登録できるようにするか」の結論）: テナントごとに種類を登録できるマスタテーブル（例: `document_types`）を用意する。各種類は「ユーザー用」か「テナント（お店）用」かを属性として持つ。
- 初期リリースでは固定 Enum で始め、のちにマスタの CRUD を追加する段階でもよい。

## 8. 非機能要件・その他

### 8.1 認可・セキュリティ

- 既存のテナント・ロール・Cedar 認可と整合させる。
- ドキュメントの閲覧・作成・更新・削除は、**4. 権限の方針**に従う（admin 全権、manager は譲渡範囲、staff は初版では未対応）。
- ファイル保存は Supabase Storage 等を使い、署名付きURLで一時アクセスさせる方式を推奨（直リンク露出を避ける）。

### 8.2 ストレージ

- ファイルアップロード先: Supabase Storage（既存 Supabase 利用と統一）を推奨。バケット・パスはテナント/ユーザーを考慮した設計とする。

### 8.3 既存アプリとの統合

- ナビ・レイアウト: 既存の Layout / テナント切替の横に「ドキュメント」または「書類管理」などのメニューを追加する想定。
- API: バックエンドに新規ルート（例: `/documents`, `/document-requirements`）を追加。フロントは Next.js の新規ページで呼び出す。

## 9. フェーズ分けの提案

| フェーズ | 内容 |
|----------|------|
| **Phase 1** | ドキュメントテーブル＋種類マスタ＋リクアイアメント定義テーブルの作成。CRUD API と「ドキュメント一覧」画面。ファイルアップロード（Supabase Storage）と有効期限の表示。 |
| **Phase 2** | 「この人は何が必要か」のクエリと API・画面。ロール別リクアイアメントの設定画面。 |
| **Phase 3** | テナント用ドキュメント（お店の書類）の明示的サポートと一覧。 |
| **Phase 4** | カレンダー表示（有効期限・支払日などの見える化）。 |

## 10. 未決・確認したい点（社長・関係者確認用）

1. **ファイル形式**: PDF のみか、画像（jpg/png）も許可するか。最大ファイルサイズの目安。
2. **「支払ったか」の管理**: カレンダーで「払ったか」を記録するなら、ドキュメントに「支払日」「次回支払予定日」などの項目を追加するか、別テーブル（支払履歴）にするか。
3. **リクアイアメントの必須化**: 未提出の場合に「アラート表示のみ」か、「ロールによる機能制限」まで行うか。
4. **1ユーザーが1テナントに複数ロール**: 現状は 1 user × 1 tenant = 1 role 想定でよいか。複数ロールの場合は「必要な要件＝全ロールの合併」とするか。
5. **manager の範囲**: ドキュメントは「テナント内なら誰のものでも manager が編集可」でよいか、それとも resource_shares のような譲渡制御を入れるか。
