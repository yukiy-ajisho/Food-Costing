Cross-tenant deprecated 実装計画（現状整理 + これからやること）
=============================================================================
作成目的: 実装着手前に「いまどこまでできているか」と「これから何をするか」を
          一枚にまとめる。

-----------------------------------------------------------------------------
1. 用語（この計画書内）
-----------------------------------------------------------------------------
・親品目 / 材料先
  recipe_lines の parent_item_id がレシピの主語、child_item_id がその行の材料として
  指す items の1件。行の種類は ingredient。
・Hide
  cross_tenant_item_shares で read が付いていない状態（共有喪失）。新規にその材料を
  指すことは別ルールで弾く。既存 ingredient 行の扱いは Hide 設計メモ参照。
・deprecated / indirect
  items.deprecated と items.deprecation_reason。indirect は連鎖・派生による廃止
  （deprecation サービスが付与）。affected の語は本プロダクトでは indirect に相当
  する説明として扱う。

-----------------------------------------------------------------------------
2. 現在の状態（すでに実装済み）— Hide まわり
-----------------------------------------------------------------------------
2.1 バックエンド（recipe-lines / items / cycle-detection-cross-tenant）
 ・他テナント prepped を材料にする read 共有の検証は、新規 ingredient 行や
   child_item_id を変える更新などに限定している。
 ・checkCycleCrossTenant は enforceShareCheck: false で呼び、共有喪失だけを理由に
   サイクル検査で失敗させない。
 ・意図: read 共有が切れたあとも、すでに存在する「材料先 A を指す ingredient 行」を、
   共有の有無だけを理由に保存のたびにエラーにしない方向。

2.2 フロント（cost ページなど）
 ・共有に基づく他テナント prepped の候補、オーナーテナント絞り込み、
   共有に載らない既存行の読み取り専用表示（Hidden by owner 系）など。

2.3 「B を軸に広げる」こととの関係
 ・消費側で prepped B を作り B のレシピに共有 prepped A を載せたうえで A を Hide しても、
   既存の A を指す行は共有喪失だけでは保存を止めない（上記）。
 ・別の親品目で自テナントの B を新たに材料として選ぶことは、B が自テナントなら
   A の共有の有無とは別問題。Hide だけでは「B を新規に選べない」にはならない。

2.4 deprecated まわりの現状（コードベース）
 ・validateRecipeLineNotDeprecated: 子 items を主に viewer 側 tenantIds で取得しており、
   他テナントの子は取れずエラーになり得る（クロステナント未対応の穴）。
 ・deprecation.ts のカスケード: 「child_item_id = 対象品目」の recipe_lines を辿り親を
   indirect にするが、クエリが渡された tenantIds に閲覧範囲がぶら下がる。同一 company
   内の他テナントのレシピまでは自動では伸びない。

3. これからやること（目標と設計原則）
-----------------------------------------------------------------------------
3.1 大目標
 ・deprecated をクロステナントの文脈でもテナント内と同系のルールで一貫させる。
 ・Hide と deprecated が重なったときは deprecated を優先する（3.3）。

3.2 カスケード（deprecation.ts）の考え方 — 仕様としての前提
 ・会社外は見ない。同一 company 内のテナントだけを対象にする。
 ・Item A が deprecated される例で考える。
 ・cross_tenant_item_shares の「いま Hide かどうか」はカスケードの前提にしない。
   いま read が付いていても、以前は付いていて A を材料にした B などがある（いま Hide
   のみ）場合もある。そのため真実は cross_tenant_item_shares ではなく recipe_lines。
 ・同一 company 内のテナントについて、child_item_id = A の ingredient 行を確認し、
   既存の deprecated カスケード（親を indirect にし再帰）と同じ解釈を、その探索範囲を
   「同一 company の全テナント」に広げて適用する。
 ・別ルールを並行で新設するのではなく、既存ロジックの「どのテナントの
   recipe_lines / items を見るか」を拡張するイメージ。

3.3 Hide と deprecated が重なったとき（deprecated 優先）
 ・ケース 1: 先に共有 → のち Hide → のち A を deprecated
   (1) deprecated 適用後、B を材料として新規にレシピを広げられない。品目プルダウンは
       既存のテナント内 deprecated と同様（例: [Deprecated] 表示・グレーアウトで選択不可）。
       自テナント候補とクロステナント候補の両方で揃える。
   (2) deprecated 前に B を材料にして C まであった場合は、テナント内と同様に B・C の
       indirect 連鎖を、3.2 の広げたカスケードで満たす。

 ・ケース 2: 先に deprecated
   read 共有の入口で選べない・載せられない（共有 UI / アクセス制御で当該行を出さない等）。
   Hide の「既存行は共有だけでは詰めない」とは別に、入口で弾く。

3.4 実装作業ブロック（計画表）
 以下の順序は仮。依存で入れ替える。

  [ ] A. 画面・API 一覧: どこで deprecated を表示・検証・弾くかを固定する。
  [ ] B. 共有候補・ケース 2: deprecated 品目を read 共有の対象・一覧に出さない（API・UI・
        必要なら RLS の確認）。
  [ ] C. validateRecipeLineNotDeprecated: 他テナントの子を、共有または正当な参照に沿って
        読み取り、deprecated を判定できるようにする（現状の tenantIds のみ取得の穴を塞ぐ）。
  [ ] D. deprecation.ts: カスケードの探索テナント範囲を同一 company 内の全テナントに拡張
        （3.2）。権限（バックエンドが他テナント items を更新してよい条件）、
        再入・パフォーマンス、undeprecate との整合を実装時に決める。
  [ ] E. フロント: クロステナント候補への [Deprecated]・グレーアウト、既存行メッセージ。
        SearchableSelect は deprecated 表示に対応済み。データ供給を揃える。
  [ ] F. 回帰: Hide 既存行（共有喪失でも保存を止めない経路）と deprecated 拒否が同じ
        リクエストでどう並ぶかのテスト・説明。
  [ ] G. cross_tenant_hide_vs_deprecated_design_notes.txt 等との矛盾解消。
