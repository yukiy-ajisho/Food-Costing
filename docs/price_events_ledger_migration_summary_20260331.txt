Price Events Ledger Migration Summary (2026-03-31)

1) Background and purpose

Food costing relies on vendor product prices being up to date.
Before this work, virtual_vendor_products stored only one mutable price value (purchase_cost),
so each update overwrote the previous value and no durable price history existed.

The goal of this work was to introduce a dynamic ledger model:
- Keep a full append-only history of price changes.
- Keep a fast current value for cost calculation.
- Prepare for two future sources of price updates:
  1) manual entry
  2) invoice/AI ingestion


2) Core design decisions

2.1 Separation of concerns
- price_events = source of truth for price history (ledger).
- virtual_vendor_products.current_price = cache/projection of latest value for fast reads.

2.2 Timing semantics
- On each inserted price event, virtual_vendor_products.current_price is updated immediately.
- virtual_vendor_products.updated_at is set to NEW.created_at (same event timestamp basis).

2.3 Event actor policy
- user_id belongs in price_events (event ownership/audit), not in virtual_vendor_products.
- This aligns with ledger modeling where "who recorded the event" is tied to the event row.


3) Implemented DB migration (first migration)

File:
- supabase/migrations/20260331110000_virtual_vendor_price_events_ledger.sql

What it does:
- Renames virtual_vendor_products.purchase_cost -> current_price
- Renames related check constraint to vendor_products_current_price_check
- Creates public.price_events table with:
  - id (uuid pk)
  - tenant_id
  - virtual_vendor_product_id
  - price
  - source_type (manual | invoice)
  - invoice_id (nullable)
  - user_id
  - created_at
- Adds indexes for tenant/product/source/invoice query patterns
- Backfills one baseline price_events row from existing virtual_vendor_products rows
  (only rows where legacy user_id is present and current_price > 0)
- Adds append-only protections (reject UPDATE/DELETE on price_events)
- Adds trigger: after INSERT on price_events, sync current_price and updated_at on vvp
- Drops legacy virtual_vendor_products.user_id FK/index/column


4) Important correction made in migration logic

Originally, trigger used:
- updated_at = now()

It was revised to:
- updated_at = NEW.created_at

Reason:
- Keep strict timestamp consistency between the event and cached row update.
- Avoid subtle mismatches when auditing or debugging event chronology.


5) Runtime issue encountered after migration and root cause

Symptom:
- Cost page returned 500 Internal Server Error on breakdown endpoint.

Root cause:
- SQL cost functions still referenced old column name purchase_cost,
  while table column had already been renamed to current_price.

Affected currently-used functions:
- calculate_item_costs_with_breakdown (GET /items/costs/breakdown uses this wrapper)
- calculate_item_costs_with_breakdown_scoped (used by missing breakdown endpoint)
- calculate_item_costs (still used by another cost route)

9) Intended model after all current DB changes

Write path (manual or invoice):
1) Insert row into price_events
2) Trigger updates virtual_vendor_products.current_price and updated_at

Read path for costing:
- Continue reading virtual_vendor_products.current_price for performance
  (without needing to compute latest event per request).

Audit/reporting path:
- Read from price_events for full chronology and source traceability.

11) Summary

This work established a ledger-first pricing architecture:
- price history is durable and append-only in price_events
- current calculation remains fast through virtual_vendor_products.current_price cache
- existing runtime breakage after rename was identified and repaired through targeted
  function reference migration.
