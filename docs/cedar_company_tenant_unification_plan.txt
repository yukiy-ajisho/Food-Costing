Cedar 統一設計（Company → Tenant）: 1つの Cedar に統合するための決定版プラン
================================================================================
作成: 2026-03
目的:
  - 既存の tenant ベース認可（food costing: items/recipes/settings）と、
    新規の company ベース認可（Team: companies/members/invitations/tenants）を、
    1つの Cedar schema + 1つの Cedar policy set + 1つの authorize 入口に統一する。
  - 統一後も「tenant ロールしか持たないユーザー（staff 等）がアクセス不能になる」事態を起こさない。

前提（確定）:
  - レイヤーは company → tenant の2階層のみ。team/department 等は増やさない。
  - 認可の情報源:
      - company ロール: company_members.role（company_admin / company_director）
      - tenant ロール: profiles.role（admin / director / manager / staff）
  - 用語（混同防止。必ずこの表記を使う）:
      - company レベル: company_members.role = company_admin, company_director
      - tenant レベル: profiles.role = admin, director, manager, staff
      - 「director」は両方にあるが、company_director（会社）と director（テナント）は別。文脈で区別する。
  - 目標とする運用（パターンB）:
      - company スコープの操作は company_members で判定
      - tenant スコープの操作は profiles（tenant membership）で判定
      - ただし Cedar は “1つ” に統一する

--------------------------------------------------------------------------------
0. 公式ベストプラクティス照合（現状の問題点を明確化）
--------------------------------------------------------------------------------

参照:
  - context の使い方（principal/resource/action の情報を context に入れない）
    https://docs.cedarpolicy.com/bestpractices/bp-using-the-context.html
  - resource creation は container に紐づけて表現する
    https://docs.cedarpolicy.com/bestpractices/bp-resources-containers.html
  - actions は API 操作ではなく業務操作に寄せる（Map actions to business domain）
    https://docs.cedarpolicy.com/bestpractices/bp-overview.html

現状の Cedar（事実）:
  A) tenant 側（backend/src/authz/*）
    - Principal(User) に tenant_id/role を属性として持たせている（これは自然）
    - create/update/delete も Resource を対象にしているが、実体が存在しない作成時に
      “collection-* / temp-*” のような擬似 Resource を使って回避している
      → 公式の「container に create 権限を付ける」推奨からズレている

  B) company 側（backend/src/authz/team/*）
    - company ロールを context.role に入れている
      → 公式の「principal の情報を context に入れない」推奨からズレている
    - Action は業務操作寄り（company::manage_invitations 等）で分かりやすい

結論（統一方針の必須要件）:
  - company ロール/tenant ロールは context ではなく “principal に紐づく情報” として扱う
    （role を1つに混ぜない。company_role と tenant_role に分離する）
  - create 権限は container（Company/Tenant）に対する action として定義する
  - actions は業務操作（capability）として定義し、必要なら action group で集約する

--------------------------------------------------------------------------------
1. 統一後の Cedar モデル（確定仕様）
--------------------------------------------------------------------------------

1.1 EntityTypes（統一 schema に入れる型）
  - User
      - id: string

  - Principal（認可主体。User と分離して role を安全に持つ）
      - id: string                (User.id)
      - company_role?: string     (company_admin/company_director。無いユーザーは属性自体を持たない)
      - tenant_role?: string      (admin/director/manager/staff。無いユーザーは属性自体を持たない)
      - tenant_id?: string        (tenant スコープ判定に必要。tenant_role がある場合のみ必須)

  - Company   （resource container）
      - id: string

  - Tenant    （resource container, Company 配下）
      - id: string
      - company_id: string   ※ もしくは parents に Company を入れて階層で表現（どちらでもよいが、統一後は1つに固定する）

  - CostResource（items/recipes/vendor_products 等の “実体リソース”）
      - id: string
      - type: string            (例: item, recipe, vendor_product, ...)
      - tenant_id: string       (所属 tenant)
      - owner_tenant_id: string (既存互換が必要なら残す)
      - item_kind?: string      (raw/prepped など)
      - user_id?: string        (作成者)

  【重要】Principal を User から分離する理由（決定）:
    - 公式ベストプラクティス上、principal と resource container を分離するほうが堅牢。
    - role は “company” と “tenant” で意味が別。1つの role に混ぜると解釈が破綻する。
    - company_role/tenant_role を optional にできるので、
      「company ロールを持たない tenant ユーザー」も自然に表現できる。

1.2 Containers（Company/Tenant を “作成権限のアンカー” にする）
  - 例:
      - company:create_tenant は Company を resource にする
      - item:create は Tenant を resource にする
    こうすることで “存在しないリソースの create” 問題を解消する。

--------------------------------------------------------------------------------
2. Actions（統一命名: 意味ベース Action に統一する）
--------------------------------------------------------------------------------

2.1 Company スコープ actions（確定）
  - company:manage_members
  - company:manage_invitations
  - company:list_tenants
  - company:create_tenant
  - （必要なら company:read を追加）

2.2 Tenant / costing スコープ actions（確定）
  - tenant:list_resources         （一覧の “list”）
  - tenant:read_resource          （単体 “read”）
  - tenant:create_item
  - tenant:update_item
  - tenant:delete_item
  - tenant:create_recipe
  - tenant:update_recipe
  - tenant:delete_recipe
  - tenant:manage_settings        （settings を1アクションに集約）
  - （vendor/products 等も同様に “create/update/delete” を type 単位で追加）

  【決定理由】:
    - CRUD を完全廃止ではなく、「意味ベースの action に確実に写像」する。
    - “settings” のように CRUD では曖昧な領域は capability を1つにする。

2.3 Action groups（運用のために必須）
  - UI では “Read/Write” のような粗い権限表示が必要な場合があるため、
    action group を schema 側で定義し、policy は group を使う。
  - 例:
      - Action::"TenantRead"   = [tenant:list_resources, tenant:read_resource]
      - Action::"TenantWrite"  = [tenant:create_*, tenant:update_*, tenant:delete_* ...]

--------------------------------------------------------------------------------
3. Policies（パターンBを崩さないための“禁止事項”込み決定）
--------------------------------------------------------------------------------

3.1 禁止事項（これをやると staff が死ぬ）
  - company ロールで tenant スコープの許可を出すこと
    例: 「company_director なら items を触れる」を書かない
  - tenant スコープは必ず profiles（tenant membership）由来の tenant ロールで判定する

3.2 Company スコープ policy（決定）
  - principal.company_role が存在し、かつ {company_admin, company_director} のときのみ permit
  - resource は Company を対象にする

3.3 Tenant スコープ policy（決定）
  - principal.tenant_role が存在し、かつ {admin, director, manager, staff} のときのみ permit
  - director（テナント）は admin と同等の権限（フルアクセス）として扱う
  - resource は Tenant（create系）または CostResource（read/update/delete系）
  - 共有（resource_shares）や prepped の例外は、現行ロジックを
    “context（request固有）” と “resource attrs” に分解して維持する

--------------------------------------------------------------------------------
4. 1つの authorize 入口（実装要件として確定）
--------------------------------------------------------------------------------

統一後の authorize は “入力正規化” を責務に持つ:
  - 入力: user_id, action, resource（Company/Tenant/CostResource いずれか）, request params
  - 正規化:
      - company スコープなら company_members から company_role を取得し Principal.company_role に入れる
      - tenant スコープなら profiles（または req.user.roles）から tenant_role と tenant_id を取得し Principal.tenant_role / tenant_id に入れる
  - Cedar に渡す:
      - entities: Principal, User, Company/Tenant, CostResource（必要な分）
      - context: request 固有だけ（例: is_shared, is_owner, time 等）

これにより:
  - “company ロールなし” のユーザーは tenant スコープでは問題なく許可される（profiles があればOK）
  - company スコープでは company_members が無いなら deny（Team の仕様どおり）

6. 仕様の最終確定（本書で確定）
--------------------------------------------------------------------------------

現状の挙動を最優先に維持するため、以下を仕様として確定する。

確定1（settings の粒度）:
  - tenant の settings は 1 capability（tenant:manage_settings）に集約する（= Yes）。
  - 目的: 現状 “settings を触れる/触れない” の大枠の挙動を崩さずに統一を成立させる。
  - 将来: settings を細分化したい要求が出た場合は、その時点で action を追加し、
          action group で UI 表示を保ったまま移行する。

確定2（costing の型分割）:
  - costing の resource は CostResource 1型で、type 属性で区別する（= No）。
  - 目的: 現状の tenant 側 Cedar が Resource 1型 + resource_type で動いているため、
          統一時に “型の分割” による大変更を避け、挙動差分を最小化する。
  - 将来: policy の可読性や監査要件で必要になった場合のみ、型別 entity（Item/Recipe 等）に分割する。
