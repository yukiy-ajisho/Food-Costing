【制約・最優先】
参考にする既存の Items ページ上のタブ（vendors タブ・base item タブなど）の機能・挙動・見た目は、本計画の実装のために一切変えてはいけない。
新規タブ・モーダル・専用ページは、既存タブを「そのまま流用して制約を足す」か、共通化する場合も既存タブの UX が変わらない形（ラッパー、別ルート、表示モードの props など）で行う。

作成: 2026-04-01
※ 本ファイルは設計メモ。実装は別タスクで行う。

================================================================================
0. スコープ
================================================================================

対象: Import invoice モーダル（主に Step 3 Match）および関連する新規ページ・子モーダル。

目的:
- Vendor / Base item / Link existing の UX・データ整合を強化する。
- インポート中に vendor や base item を追加・修正できる導線を用意する。

================================================================================
1. 現状確認（実装前のベースライン）
================================================================================

1-1. Base item プルダウン（実装前のベースライン記録）
- 当時のコードでは、候補はテナントの baseItems 全体で、vendor では絞り込んでいなかった。
- 本計画の途中案では「vendor に紐づくものだけ」への変更も検討された。
- **実装の確定内容はセクション 7 を正とする**（New はテナント全件、Existing は vendor 絞り）。

1-2. Vendor 未選択時
- Link existing は「vendor を選んで」と無効化されている。
- Base item は vendor なしでも選べる状態（要変更: Link existing と同様に無効化）。

================================================================================
2. 機能要件（確定イメージ）
================================================================================

2-1. Vendor プルダウン周り
- Match ステップで、vendor プルダウンの右に「+ New vendor」相当のボタンを置く。
- クリックで新しいブラウザタブを開く（window.open 等）。同一オリジンの専用 URL。
- 親タブとの同期:
  - モーダル全体の再マウントや全データ再取得はしない。
  - vendor プルダウン用のリストだけ更新し、新規作成された vendor が選択肢に現れるようにする。
  - postMessage / BroadcastChannel 等で origin 検証を行う（セキュリティ）。
- 認証: 新規タブは通常どおり Cookie 共有で同一セッション想定。

2-2. New vendor 専用ページ（新規タブ）
- 表示: サイドバー・ヘッダー・Items 上のタブ切替は出さない。vendors のテーブル＋編集体験のみ。
- 既存 vendors の一覧の下に、あらかじめ 1 行の空白レコードがある（新規入力用）。
- 「Add new vendor」のような追加ボタンは表示しない（空白行が新規枠）。
- ゴミ箱（削除 UI）は出さない。
- 同時に複数の新規追加フローは想定しない（1 枠）。
- ボタンは最初から Save と Cancel のみ。
- Save 完了までタブは閉じない。Save 中は Save を無効化。
- Save 成功後にタブを閉じる（window.close 等。ブラウザにより制限あり得る旨は実装時に注記）。
- URL をユーザーが変えられないことは保証できないが、専用の固定パスで提供する。

2-3. Vendor 変更の連動
- ブロックの vendor が変わるたびに:
  - Base item プルダウンの候補・選択の妥当性を更新する。
  - Link existing プルダウンの候補・選択の妥当性を更新する（既存ロジックを拡張）。
- 同期・候補の再計算などでプルダウン内容が差し替わる間は、該当セルでプルダウンを一時的に非表示にし、「loading...」を表示する:
  - Match が New の行 → Base item 列のプルダウン側。
  - Match が Existing の行 → Link existing 列のプルダウン側。
- 例: ヒントで「ja」にマッチしていたが、新規 vendor「new」を作ったあと、既に選んでいた base item / link existing が新 vendor に存在しなければ無効化・クリア等（仕様どおり整合を取る）。

2-4. Base item / Link existing（候補の絞り込み）
- **確定仕様はセクション 7**。
- **New** 行の Base item: テナントの baseItems 全件（vendor では絞らない）。vendor 未選択時は Base item を選べない（Link existing と同様の UX）。
- **Existing** 行の Link existing: 選択中 vendor の仕入れ商品（VVP）のみ（vendor で絞る）。vendor 未選択時は選べない。

2-5. 非質量単位と Base item 候補の赤ポチ（New モード・プルダウン各行）
- 前提: 請求から読み取った当該行の purchase_unit（product の単位）が「非質量」である場合にのみ、この赤表示ルールを適用する。
- 請求行の単位が質量（例: g, kg などアプリ上の質量扱い）の場合: Base item プルダウン内の**どの候補行にも赤ポチは付けない**。
- 請求行の単位が非質量の場合: Base item プルダウンを開いたとき、**候補のうち specific weight（specific_weight）が未設定の base item は非質量に使えない**ため、その**各行に赤ポチ**（赤いインジケータ）を付ける。**specific_weight がある行には赤ポチは付けない**。
- Next 進行可否: 非質量単位の行で、base item 未選択、または選択中の base item に specific_weight がない場合は進行不可とし、英語のポップアップで伝える（実装時に文言・条件を既存の単位定数と整合）。
- 請求が質量単位なら、この赤ポチ行表示・このブロック用の Next ブロックは発生しない（または該当しない）。

2-6. Base item クイック編集モーダル（赤状態時の「Edit」テキスト）
- 上記 2-5 の文脈で問題となる選択（例: 赤ポチ付き候補が選ばれている、または未選択で Next 不可）に応じて、Base item プルダウンの右側に「Edit」テキストリンクを表示する（以前仕様の「赤のものが選ばれていたら」に相当する条件で実装時に確定）。
- クリックでモーダルを開く。
- 中身は Base item タブと同じテーブル構成・編集体験だが、表示レコードは現在選ばれている base item のみ。
- Add はない。
- 開いた時点で「Edit を押した直後」と同じ編集モード。Base item 名は変更不可。
- ボタンは Save / Cancel のみ（Edit ボタンは出さない）。
- Save 完了まで閉じない。Save 中はボタン無効化。

2-7. + New base item
- 配置: Match ステップのテーブルで、「Base item」列のヘッダー（ラベル）の右側に「+ new base item」相当。
- ドロップダウンの横ではなく、列ヘッダー側。
- 新規 vendor がタブ違いであるのに対し、こちらは新規ブラウザタブではなくアプリ内モーダル。
- モーダル内容: Base item タブと同じテーブル構成。既存行の下に空白の新規行が 1 つある状態から始める。
- Add new base itemのボタンは必要ない。
- ボタンは Save と Cancel。キャンセルでモーダルを閉じる。
- Save は完了後にモーダルを閉じる。Save 中は Save 無効化。
- 保存後、親はモーダル全体ではなく Base item プルダウン（および必要なら候補集合）だけ更新し、新規 base item が選べるようにする。

================================================================================

- 認証: 専用ページ・モーダルとも既存の認証フローに乗せる。
- セキュリティ: タブ間連携は postMessage 等で origin チェック。noopener 付与。
- 見た目: vendors / base item は items ページの該当タブと同一コンポーネントまたは共通化してドリフトを防ぐ。

4. 実装計画（ステップ分割）
================================================================================

ステップ A — データ連動（インポートモーダルのみ）
  A1. vendor 未選択時は Base item プルダウンを無効化（Link existing と同様の文言・挙動）。
  A2. Base item / Link existing の候補ルールを確定する（**セクション 7**: New はテナント全件、Existing は vendor 絞り）。
  A3. vendor 変更時に base_item_id / linked_vvp_id を既存ルールに沿ってクリア・再提案する（必要なら拡張）。
  A4. vendor 変更・同期でプルダウン内容が更新される間、該当 Match モードのプルダウンを隠し「loading...」を出す（New → Base item、Existing → Link existing）。

ステップ B — 検証と UI フィードバック（New モード）
  B1. 請求行が非質量単位のときのみ、Base item プルダウン候補のうち specific_weight 未設定の行に赤ポチ。質量単位のときは候補行に赤ポチなし。
  B2. 非質量で未選択または不適合な base item のとき Next 不可・英語アラート。
  B3. 赤のときだけ Base item 右に「Edit」リンク（モーダルはステップ C とまとめても可）。

ステップ C — Base item クイック編集モーダル
  C1. items の base item テーブル＋編集を単一レコード・名前固定・Save/Cancel のみにしたラッパコンポーネントを切り出しまたは再利用。
  C2. InvoiceImportModal から開閉・保存後に baseItems（または該当スコープ）だけ更新。

ステップ D — + New base item モーダル
  D1. 列ヘッダー右にボタン配置。
  D2. Base item タブ相当 UI のモーダル（下に空白 1 行、Save/Cancel、保存中無効）。
  D3. 保存成功後、親の base item 候補と必要なら選択値を更新。

ステップ E — + New vendor 専用ページ
  E1. ルート追加（例: /items/vendors-embed 等）とミニマル layout（サイドバー・タブ非表示）。
  E2. vendors テーブル共通化。最下行に空白 1 行、Save/Cancel のみ、削除・追加ボタンなし。
  E3. Match の「+ New vendor」から window.open。保存成功後タブを閉じる処理と、親へのメッセージで vendor リストのみ更新。

ステップ F — 結合テスト・ポリッシュ
  F1. vendor 切替・新規 vendor・新規 base item・赤検証・Link existing の組み合わせを手動確認。
  F2. アクセシビリティ（フォーカス、新規タブの案内）とエラーハンドリング。

記載日: 2026-04-01（以降の実装反映）

上記 2-4・1-1 では「選択中 vendor に紐づく base item のみ」を New のプルダウン候補とする方向があったが、

  テナント境界でのみ絞る。vendor では絞らない）。
- Match が **Existing** の行: **Link existing** は従来どおり **選択中 vendor の仕入れ商品（VVP）のみ**
  を候補とする（vendor で絞る）。

================================================================================
8. 追記（2026-04-01）— New 時の base item  suggest と Existing→New 切替
================================================================================

8-1. New 時の Base item（suggest は任意・無理に選ばない）
- Match が **New** のとき、Base item プルダウンは **テナントの base item 全件**を候補とする（セクション 7 どおり）。
- **product name（等）からの suggest**（既存の `buildBaseItemSelectOptions` / `suggestBaseItemIdFromProductName` の思想）により、
  **十分な候補があるときは 1 件を初期選択**してよい。
- **該当する base item が suggest できない／信頼できない場合は、無理に 1 件を選ばない**（`base_item_id` は空のままでよい）。
- **非質量単位の赤ポチ（warningDot）・Next 進行条件**など、既存の New モード向けルールは、選択がある場合・未選択の場合それぞれに応じて踏襲する。

8-2. Existing から New へユーザーが切り替えたとき
- **Link existing** で選ばれている VVP が実際の意図と合わない、などの理由で、ユーザーが **Match を Existing から New に切り替えた**場合:
  - その瞬間、**Base item 列のプルダウン**を通常の New モードと同様に扱う（候補はテナント全件、vendor 未選択なら無効化など、セクション 7・2-4 と整合）。
  - **product name（等）を用いて base item の照合・suggest を行い、該当があればプルダウン上で選ばれた状態にできる**（8-1 と同じく、**無理に選ばなくてよい**）。

8-3. 手動での New ↔ Existing 切り替え（どちらに実装の重点があるか）
- **New → Existing**（ユーザーがラジオで Existing にしたとき）:
  - 自動判定で **New** になっている行は、多くの場合 **当該 vendor の VVP で `score > 0` となるものが無かった**（または候補ゼロ）ためである。
  - その状態で Existing に切り替えても **マスタの VVP は増えない**ので、**自動で適切な `linked_vvp_id` が付くとは限らない**（**空のまま**になり、**ユーザーが Link existing で選ぶ**のが自然なことが多い）。実装・UX の重点は **base item ではなく Link existing** 側である。
- **Existing → New**（ユーザーがラジオで New にしたとき）:
  - **こちらが base item 候補の列挙と product 由来の suggest を必ず走らせるべきケース**である。`linked_vvp_id` をクリアし、**テナントの有効 base 全件**（9-6 の deprecated 除外）＋ **8-1** の suggest ルール（無理に 1 件にしない）を適用する。

8-4. 関連（以前の合意の整理）
- ブロックの **vendor 変更**時は、**再マッチ**により **New / Existing が変わりうる**（別メモの「vendor 変更で match 再実行」と整合）。
- 本節 8 は、**ユーザー操作による Existing→New** においても、**自動 vendor 変更** においても、**New になった行は Base item UI・suggest ルールを一貫して適用する**、という趣旨とする。

================================================================================
9. 追記（2026-04-01）— VVP マスタ照合による New/Existing 初期化・vendor 再マッチ（実装依頼の要点）
================================================================================

9-1. A. 「VVP が見つかる／見つからない」で New / Existing を決める
- **対象**: Match（ステップ 3）に入った**あと**、かつ**そのブロックに vendorId がある**ときの**各行**。
- **候補集合**: 当該 vendor の **非 deprecated の VVP のみ**（deprecated は含めない）。
- **判定**: 請求行（product / brand / qty / unit など）に対し、既存の **`scoreVvpForInvoiceRow` やランキングに近い考え方**でスコア付けし、「十分マッチした」と言えるかを見る（**閾値は 9-6**）。
- **十分マッチする VVP がある** → 初期は **existing**。`linked_vvp_id` をその**ベスト**に（ユーザーはあとからプルダウンで変更可）。
- **ない**（候補ゼロ、またはスコアが閾値未満など） → 初期は **new**。`linked_vvp_id` は**空**。
- **new** のときの Base item: プルダウンに **テナントの base item 全部** ＋ **product name からの suggest** で **可能なら 1 件選択**（`buildBaseItemSelectOptions` / `suggestBaseItemIdFromProductName` の思想。非質量の warningDot 等はセクション 2-5・8-1）。
- **現状の整理**: パース直後に**常に** `matchMode: "new"` だけ、という挙動はやめる。**マスタ照合の結果**でステップ 3 における**初期の New / Existing** を決める。

9-2. B. ベンダー変更のたびに「もう一度マッチ」
- ブロックの **vendorId が変わるたび**（ユーザーがセレクト変更）、そのブロック内の**全行**について **9-1 と同じ判定**をやり直す。
- **その結果**、`matchMode` は **new ↔ existing** に**変わりうる**。
- **existing** になったら: `linked_vvp_id` を**新 vendor** のランキングで**付け直す**（既存の vendor フィルタはそのまま活かす）。
- **new** になったら: `linked_vvp_id` を**空**にし、`base_item_id` は **再度 suggest** する。**「既に入っていたら触らない」ではなく**、ベンダーを変えたら**作り直す**方向（「再び match」と一致する理解）。

9-3. 例（詳しく・複数）
- **前提**: マスタに Vendor JFC, Vendor A、base item BULLDOG SAUCE などがある。

**例 1** — JFC に BULLDOG の VVP がある（きれいに一致）
- 請求: ブロックのヒント JFC、行 product = "BULLDOG SAUCE"。
- ステップ 3: vendorId が JFC に決まる（ヒント or ユーザー選択）。
- JFC の VVP に請求行と**強く一致する**行がある（名前・数量・単位などでスコア十分）。
- **期待**: その行は **existing**、`linked_vvp_id` はその VVP。Base item 列は existing では**非表示／未使用**（現 UI どおり）。Base item の「全部＋ suggest」は **new のときの話**なので本例では主に触れない。

**例 2** — JFC はいるが、BULLDOG の VVP が JFC に無い
- 請求: JFC, product = "BULLDOG"。マスタ: JFC に BULLDOG の VVP は無い。ステップ 3: vendorId = JFC。
- **期待**: 候補ゼロ or スコア不足 → 行は **new**。`linked_vvp_id` **空**。
- Base item: プルダウンは**全 base item**。"BULLDOG" に部分一致する base 名があれば **suggest で 1 件選択**（無理に選ばないルールは 8-1）。

**例 3** — 例 2 のあと、ベンダーを Vendor A に変更（A に BULLDOG の VVP がある）
- 変更前: **new**、base item は仮に X が選ばれていた。Vendor A には BULLDOG と**強く一致する** VVP がある。
- **期待（再マッチ）**: 行は **existing** に切り替わる。`linked_vvp_id` = その VVP。`base_item_id` は **空にするか**、少なくとも **new 用の選択は無効化**（UI は existing なので **Link existing** だけ、現 UI に合わせる）。
- もし A にも BULLDOG が**全く無い**なら、再マッチ後も **new** のまま、base item を product から**再度 suggest**（以前の X は**上書きしてよい**、という理解）。

**例 4** — 最初 Existing でリンク済み → ベンダーを別店（例: JFC）に変えた
- 行は **existing**、`linked_vvp_id` = Vendor A の VVP1。ユーザーがブロックのベンダーを JFC に変更。JFC には請求行に似た VVP が**無い**。
- **期待**: 再マッチの結果 **new** に落ちる。`linked_vvp_id` **クリア**。base item を product から**再度 suggest**、プルダウンは**全件**。

**例 5** — 同じベンダー内で「弱い一致」だけ複数ある
- vendorId 固定。VVP は複数あるが、product 名の一致が弱い／スコアがギリギリ。
- **確定（9-6）**: ベストのスコアが **`score > 0`** なら **existing**、ベストを自動選択（ユーザーがプルダウンで変更可）。**`score === 0` のみ**なら **new**（候補ゼロ扱いに近い）。

**例 6** — ステップ 2 で product を直したあと、ステップ 3 に入る
- パースでは誤認識。Review で product を修正してから Next。
- **期待**: ステップ 3 の**初回マッチ**は**修正後の product** を使う（suggest / ランキングの入力は行データなので、自然にそうなる想定）。

9-4. まとめ（理解の一文）
- **「VVP が請求＋そのブロック vendor で十分見つかる → existing＋リンク、見つからない → new＋base 全件＋suggest」** を、**ステップ 3 で明示的に**行う（パース時の**一律 new はやめる／上書きする**）。
- **vendor 変更のたびに**同じ判定を**全行**に再実行し、**new / existing が変わりうる**し、**紐付け・base も作り直す**（9-2）。

9-5. 再マッチのトリガー（`vendorProducts` 差し替えのみは対象外）
- **自動再マッチを行うのは**、次に**限定**する。
  - ブロックの **vendorId が変わったとき**（9-2）。
  - **ステップ 3 に入り**、当該ブロックの **vendor が確定したあとの初回マッチ**（vendor ヒントからの自動選択を含む。9-1・例 6 と整合）。
- 親から `InvoiceImportModal` へ渡る **`vendorProducts` が refetch 等で配列が差し替わっただけ**で、**ユーザーが選んでいる vendor は変わっていない**場合、**それ単体をトリガーにした自動再マッチは行わない**（実装スコープから除外する）。

9-6. 実装で確定したパラメータ・Base item 候補
- **VVP マッチ閾値**: 「十分マッチして **existing** にできるか」は、当該 vendor の非 deprecated VVP のうち **ランキング上位のスコアが `score > 0`** かどうかで決める。**より厳しい閾値は設けない**（現行の `scoreVvpForInvoiceRow` と同じ考え方のまま）。
- **Base item プルダウン（New 行）**: **`base_items.deprecated` が NULL の行のみ**を候補に含める（**deprecated は候補から除外**。現状通り）。データは親の取得結果を前提とし、モーダル内でも **deprecated 行を除外する**など二重に担保してよい。
