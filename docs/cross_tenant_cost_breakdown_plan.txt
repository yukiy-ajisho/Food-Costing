================================================================================
Cross-tenant 材料を含む原価 breakdown の計画
================================================================================
目的: 同一 company 内で共有された他テナントの prepped を材料にしたとき、
      Cost 画面の cost/g・行 Cost・親の COG/LABOR（および each yield まわり）が
      自テナントのみのレシピと同様に一貫して表示・計算されるようにする。

制約:
  - 既存の calculate_item_costs_with_breakdown(p_tenant_id) の挙動
    （共有なし・親子とも同一テナント）は壊さない。
  - 他テナント data へのアクセスは「共有が証明できる場合」の例外に限定する。
  - 本書は設計・手順の記録のみ。実装は別タスクで行う。

関連ファイル（参照）:
  - supabase/schema_dump.sql … calculate_item_costs_with_breakdown
  - backend/src/routes/cost.ts … GET /items/costs/breakdown
  - backend/src/routes/cross-tenant-item-shares.ts … 共有一覧・available
  - supabase/migrations/*_cross_tenant_item_shares.sql … 共有テーブル
  - frontend/app/cost/page.tsx … breakdown 表示、convertToGrams、材料 Cost 列

--------------------------------------------------------------------------------
1. 背景（いま起きていること）
--------------------------------------------------------------------------------
  - ビジネス上の計算式（子の $/g × 投入 g、yield で割る、労務を足す）は
    材料の所有者が別テナントでも変わらない。
  - 一方、現行 RPC は「作業用の原価表 temp_item_costs を、原則その
    p_tenant_id の raw / prepped だけで埋める」「prepped 積み上げで
    child_items.t tenant_id = p_tenant_id を要求」など、単一テナント閉じ
    の前提がある。
  - フロントも材料行の解決に availableItems（自テナント）のみを使う箇所が
    あり、breakdown のキーもテナント B の RPC 結果に他テナント item_id が
    載らないため UI が「-」や 0g になりうる。

--------------------------------------------------------------------------------
2. 目標（完了の定義）
--------------------------------------------------------------------------------
  - テナント B のレシピがテナント A の共有 prepped（read 許可済み）を材料に
    含む場合:
    (a) GET /items/costs/breakdown（tenant = B）の結果に、B 自身の prepped
        の food/labor/total $/g が、当該材料のコストを反映して算出される。
    (b) 材料行: cost/g（prepped の food 相当）と行 Cost が表示される。
    (c) 親の yield が each のとき、g/each および COG/LABOR の表示が、
        材料グラム換算が成立する範囲で自テナントと同様に意味を持つ。
  - 共有が無い・非対象の他テナント item を参照する recipe 行は、方針を決めて
    扱う（エラー扱い・NULL・ログ等）。無断で他テナント原価を参照しない。

--------------------------------------------------------------------------------
3. 設計方針（ベストプラクティスとの整合）
--------------------------------------------------------------------------------
  - 計算式そのものは変更しない。変更するのは「temp_item_costs に載せる子の
    原価を、どの条件で、どのテナント文脈から補うか」という段取りのみ。
  - 単一関数 calculate_item_costs_with_breakdown を拡張し、
    「通常パス」と「クロステナント例外パス」を条件で明確に分離する。
  - 例外パスに入る条件の例（最終仕様で厳密化）:
    · recipe_lines は tenant_id = p_tenant_id（閲覧側）の ingredient 行。
    · child_item_id に対応する items.tenant_id ≠ p_tenant_id。
    · cross_tenant_item_shares で、当該 item・owner_tenant・company・
      target（company 全体 or 閲覧 tenant）および allowed_actions に read が
      含まれる等、既存 API と同等の認可を SQL 内で再検証する。
  - パフォーマンス: cross_tenant_item_shares と items の突き合わせに効く
    インデックスを確認・追加検討する。
  - 循環依存: テナントをまたいだ A→B→A のような材料グラフは、禁止・検出・
    深さ制限のいずれかを設計で決める。

--------------------------------------------------------------------------------
4. バックエンド（SQL / RPC）タスク
--------------------------------------------------------------------------------
  4.1 現行関数の流れの再確認（退行防止のため）
       - ステップ1: raw を p_tenant_id で seed
       - ステップ2: prepped を temp_ingredient_edges と temp_item_costs の
         INNER JOIN で順次 INSERT、child_items.tenant_id = p_tenant_id 等
       - ステップ3: 返却行は i.tenant_id = p_tenant_id のみ

  4.2 例外パス（案）
       A) 「外国の prepped」で、共有が有効な child_item_id について、
          そのアイテムを **オーナーテナント（items.tenant_id）を
          p_tenant_id として** 再計算するのと論理同等な food/labor $/g を
          得て temp_item_costs に INSERT する。
          - 実装選択: 同一関数内でサブルーチン相当の SQL ブロックに分割、
            または「子グラフのみ owner テナントで同型の計算」を再利用。
          - 重要: 無共有の foreign id は載せない。
       B) 既存のステップ2 ループにおいて、child_items.tenant_id = p_tenant_id
          の条件を「共有済み foreign child」のときは緩和し、かつ tc に
          外国子の行が存在することを保証する（A で先に seed 済みにする）。

  4.3 返却仕様
       - 返す out_item_id は引き続き「p_tenant_id に属する items」のみで
         よい（現 UI は tenant スコープの breakdown を想定）。
       - 外国アイテムの行を返す必要はない（材料行表示は別補完でも可だが、
         親の数値が正しければ最低限は満たせる）。

  4.4 マイグレーション
       - CREATE OR REPLACE FUNCTION でデプロイ。
       - 既存の migration 方針に合わせ、新規 migration ファイルに全文を置く。

  4.5 テスト（推奨）
       - 共有なし・従来レシピのみ: 現行と同じ数値（ゴールデン）を保持。
       - 共有あり・B が A の prepped を1材料に使用: 親 B の breakdown が
         期待通り。
       - 共有無効で foreign id を参照: 定めた失敗／スキップと整合。

--------------------------------------------------------------------------------
5. フロントエンド（必要に応じた追従）
--------------------------------------------------------------------------------
  - breakdown が正しければ親の COG/LABOR は改善するが、材料行の cost/g は
    依然「childItem を availableItems で見つける」前提だと - のまま残る
    可能性がある。
  - 計画: crossTenantAvailableItems（または共有 API が返す each_grams 等）を
    参照し、自テナントに無い child_item_id でも prepped の
    costBreakdown[child_id] を参照できるよう表示ロジックを揃える。
  - convertToGrams 等、材料グラム合計が 0 になる経路も同様に共有メタデータ
    との突き合わせを検討する。

--------------------------------------------------------------------------------
6. API・セキュリティ
--------------------------------------------------------------------------------
  - breakdown は認証済み・X-Tenant-ID（閲覧テナント）付きで呼ばれる。
  - RPC 内の共有条件は、アプリの cross-tenant-item-shares と矛盾しないよう
    ドキュメント化する。
  - 監査: 不要と判断するか、将来の要件で検討。

--------------------------------------------------------------------------------
7. 実装順序の提案
--------------------------------------------------------------------------------
  1) SQL: 共有条件を満たす foreign prepped のみ temp_item_costs へ seed する
     ブロックを設計・実装し、ステップ2 の条件緩和と整合。
  2) ローカル／Supabase で既存ケースとクロスケースの手動または自動検証。
  3) フロント: 材料 Cost 列・グラム換算のデータ源を例外対応。
  4) 回帰: Cost ページ主要シナリオ、パフォーマンス目視。

--------------------------------------------------------------------------------
8. オープン課題（実装前に決める）
--------------------------------------------------------------------------------
  - 共有 prepped がさらに「共有された材料」を含むネストの深さ・再帰の扱い。
  - foreign raw を材料にする要件の有無（現状 cross は prepped のみ共有の
    想定なら対象外でよいか）。
  - breakdown に載らない外国 item を UI でどう表示するか（-/説明文）。
